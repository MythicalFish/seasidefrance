# Brainstorm

Use this skill before scoping or planning when the request would benefit from deeper ideation.

## Workflow

1. Read the most relevant repo context first such as `README.md`, `specs/`, issue notes, or nearby code.
2. Restate the problem in plain language and the outcome the user likely wants.
3. Identify the real design surface: UX, architecture, state, data flow, and verification concerns.
4. Generate 2-4 realistic approaches when trade-offs exist.
5. Evaluate each approach concretely with likely touch points, benefits, risks, and complexity.
6. Recommend one approach unless the choice is genuinely open.
7. End with a focused approval question or a small set of decisions.

## Rules

- Do not jump straight from ambiguity to implementation details.
- Prefer specific trade-offs over generic pros and cons.
- Use concrete file and system references whenever possible.
- If one option is clearly better, say so.
- Keep the conversation collaborative and easy to react to.
